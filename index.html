How if/else works
1
The decision flowchart
if checks a condition. If it's true, run the then block. If false, run the else block (optional). Either way, code continues after end.
Pattern

if condition then ... elseif other condition then ... else ... end. You can chain as many elseif as you need. else is a catch-all fallback.

ServerScriptService › Script
local hp = 30

-- basic if/else
if hp > 0 then
    print("Player is alive")
else
    print("Player is dead")
end

-- chained elseif
if hp >= 75 then
    print("Healthy")
elseif hp >= 30 then
    print("Damaged")
elseif hp > 0 then
    print("Critical!")
else
    print("Dead.")
end

-- using and/or in conditions
local hasKey = true
if hp > 0 and hasKey then
    print("Can open the door")
end

Read this first — then write the script
How this script works

TweenService smoothly moves a Part from one position to another over time. CFrame represents both position and rotation. script.Parent is the door Part itself. The isOpen flag (debounce) prevents the door triggering again while it's already moving.

2
The door script
Part › Script
local TweenService = game:GetService("TweenService")
local door         = script.Parent
local isOpen       = false          -- debounce: prevents double-triggering

local closedPos    = door.CFrame
local openPos      = door.CFrame * CFrame.new(0, 8, 0)  -- 8 studs up

local info = TweenInfo.new(0.6, Enum.EasingStyle.Quad)

local function openDoor()
    if isOpen then return end        -- already open? stop here
    isOpen = true

    TweenService:Create(door, info, {CFrame = openPos}):Play()
    task.wait(3)                       -- stay open 3 seconds

    local close = TweenService:Create(door, info, {CFrame = closedPos})
    close:Play()
    close.Completed:Wait()            -- wait for close animation
    isOpen = false
end

door.Touched:Connect(function(hit)
    if hit.Parent:FindFirstChild("Humanoid") then
        openDoor()
    end
end)

Roblox automatically displays any Folder named exactly "leaderstats" inside a player as the scoreboard. You create the folder in code when each player joins. Any IntValue or NumberValue inside it shows as a leaderboard row.

1
Create the leaderboard on join

leaderstats-explorer

RESULT

Explorer shows a folder called leaderstats inside the player, with a Coins IntValue inside. Top-right scoreboard appears automatically.

ServerScriptService › Script
local Players = game:GetService("Players")

Players.PlayerAdded:Connect(function(player)
    local stats = Instance.new("Folder")
    stats.Name   = "leaderstats"  -- exact spelling, case-sensitive!
    stats.Parent = player

    local coins = Instance.new("IntValue")
    coins.Name   = "Coins"
    coins.Value  = 0
    coins.Parent = stats
end)

03If it doesn't appear, check the folder name is spelled exactly "leaderstats" (lowercase l, lowercase s).
Part 2 — Coin Part
How the coin script works

GetPlayerFromCharacter converts a character model back into the Player object — that's how you reach the leaderboard. Transparency = 1 hides the coin without destroying it, so you can show it again after 5 seconds.

2
Build the coin Part, add this Script inside it

coin-part

BUILD IT

Insert Part → small, round-ish, yellow. Name it "Coin" in Explorer. Right-click → Insert Object → Script.

Coin Part › Script
local coin      = script.Parent
local collected = false

coin.Touched:Connect(function(hit)
    if collected then return end

    local char   = hit.Parent
    local player = game.Players:GetPlayerFromCharacter(char)
    if not player then return end   -- not a player, ignore

    collected                                = true
    player.leaderstats.Coins.Value          += 1
    coin.Transparency                         = 1
    coin.CanCollide                           = false

    task.wait(5)                              -- respawn after 5 seconds
    coin.Transparency                         = 0
    coin.CanCollide                           = true
    collected                                = false
end)

coin-collected

RESULT

Walk over the coin → disappears → leaderboard shows Coins: 1. Reappears after 5 seconds.

@RoDevAcademy · UPLOADING SOON

Transcript will be added when the video uploads.
Look at Explorer. See how items are indented underneath other items? Everything lives inside something else. That nesting is called hierarchy, and it's how Studio organizes everything.
Parent and child
→
The simple idea
Parent = the thing that contains.
Child = the thing that is contained.

Example: Workspace is the parent. Your Part is a child of Workspace.

In Explorer: children are indented under their parent. Click the arrow next to any item to see what's inside it.
1
See it in Explorer
In Explorer, click the arrow next to Workspace to expand it.

See Baseplate and your Part indented below it? Those are children of Workspace.

If Workspace is the parent, and you delete Workspace — everything inside it disappears too.
Workspace → Baseplate is a parent → child relationship.
2
Why this matters for scripting
When you add a Script inside a Part later, the Script is a child of the Part. In Lua code you access the Part by writing script.Parent.

That's how a script finds the object it belongs to — it just looks at its parent. You'll use this in Lesson 3.

TABLES
& ARRAYS

YOU'LL BUILDA list of player names you can loop through
Why this matters

A table is the only collection type in Lua. Every list of items in Roblox — players, checkpoints, weapons — is a table. Master this and data becomes easy.

Read first
The concept

A table stores multiple values in one place. You can use it as a numbered list (array) or as a key→value dictionary.

arrays.lua
-- Array (numbered list)
local players = {"Alex", "Maya", "Zed"}

-- Access by index (starts at 1!)
print(players[1])  -- Alex
print(players[3])  -- Zed

-- Loop through with ipairs
for i, name in ipairs(players) do
  print(i.. ". " .. name)
end
dictionary.lua
-- Dictionary (key → value)
local stats = {
  coins = 50,
  level = 3,
  name  = "Alex"
}

print(stats.name)    -- Alex
print(stats.coins)   -- 50
stats.coins = stats.coins + 10
print(stats.coins)   -- 60

SCOPE &
LOCAL VS
GLOBAL

YOU'LL UNDERSTANDWhy variables disappear and how to fix it
Why this matters

Scope is why a variable defined inside a function can't be seen outside it. Always use local. Global variables cause bugs that take hours to find.

scope.lua
local outerVar = "I'm outside"

local function myFunc()
  local innerVar = "I'm inside"
  print(outerVar)   -- works: outer is visible inside
  print(innerVar)   -- works
end

myFunc()
print(outerVar)     -- works
print(innerVar)     -- nil! innerVar doesn't exist here

C12 · Debugging

DEBUGGING
& ERRORS

YOU'LL LEARNHow to read errors, use pcall, and find bugs fast
Why this matters

Every developer spends more time fixing code than writing it. The faster you can read an error and find the problem, the faster you ship games.

Reading errors

An error message looks like: Script:4: attempt to concatenate a nil value
Script:4 = the file and line number. "concatenate a nil value" = you used .. on something that was nil.

pcall.lua
-- pcall = protected call. It catches errors instead of crashing.
local ok, err = pcall(function()
  error("something went wrong")
end)

print(ok)   -- false (it failed)
print(err)  -- the error message

-- In Roblox you use pcall for DataStore calls:
-- local ok, data = pcall(function() return DataStore:GetAsync(key) end)
-- if not ok then print("DataStore failed: " .. data) end

Why this matters

Every UI element in Roblox — buttons, labels, health bars — lives inside a ScreenGui. Without it, nothing shows on screen. The ScreenGui is your canvas.

The hierarchy

StarterGui → ScreenGui → Frame → buttons, labels etc.
Everything nested inside ScreenGui gets copied to each player when they join.

Access UI from a LocalScript

U03 · Click Events

MOUSE
BUTTON1
CLICK

YOU'LL BUILDA click counter — every click adds 1 and prints the total
Why this matters

MouseButton1Click is the event that fires when a player clicks a TextButton. This is the most used event in all of UI scripting — shops, menus, clicking games — they all start here.

MouseButton1Click = left click, released. MouseButton1Down = fires on press. MouseButton2Click = right click. For buttons you almost always want MouseButton1Click.

Button click event pattern
local button = frame:WaitForChild("ClickButton")
local label  = frame:WaitForChild("CoinLabel")
local coins  = 0

button.MouseButton1Click:Connect(function()
  coins = coins + 1
  label.Text = "Coins: " .. coins
end)

U05 · Communication

REMOTE
EVENTS

YOU'LL UNDERSTANDHow the client tells the server "something happened"
Why this matters

A RemoteEvent is the bridge between client and server. The client fires it. The server listens. Without this, your UI can't affect the leaderboard or save data. Every real Roblox game uses this.

Client fires → Server listens
-- CLIENT (LocalScript in ClickingGui):
local event = game.ReplicatedStorage:WaitForChild("AddCoin")

button.MouseButton1Click:Connect(function()
  event:FireServer()  -- sends message to server
end)

-- SERVER (Script in ServerScriptService):
local event = game.ReplicatedStorage:WaitForChild("AddCoin")

event.OnServerEvent:Connect(function(player)
  -- player object passed automatically
  player.leaderstats.Coins.Value = player.leaderstats.Coins.Value + 1
end)
